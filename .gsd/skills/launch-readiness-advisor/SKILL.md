# Launch Readiness Advisor

**Primary use:** Tiramisup should use this skill to generate user-facing launch readiness guidance, blocker summaries, and staged release recommendations.

Use this skill when recommending how users should prepare for launch, separate critical blockers from non-critical remaining work, and understand what should still be visible after launch. This skill is for **launch sequencing and readiness**, not for store-policy or analytics setup.

## Use This For

Apply this skill when the task involves any of the following:
- launch readiness summaries
- blocker prioritization
- pre-launch checklist interpretation
- “what still blocks launch” explanations
- post-launch residual work visibility
- release sequencing and stage transitions

## Do Not Use This For

- App Store / Play Store policy or submission guidance
- ASO or listing optimization
- analytics instrumentation or event schema design
- generic strategy advice that is not tied to launch readiness

## Core Principle

Launch readiness should be **stage-aware and brutally clear**. The user must know what blocks launch, what can wait, and what should remain visible after launch.

## Required Recommendation Areas

When creating launch readiness guidance for users, make sure recommendations cover these groups.

### 1. Critical Blockers

Always check whether the user has:
- unresolved high-priority items
- broken core flows
- missing essential content or setup
- readiness gaps that would confuse users or reviewers

### 2. Non-Critical Remaining Work

Always make room for:
- polish items that do not block launch
- items that should stay visible after launch
- follow-up work that improves quality without delaying release

### 3. Stage Transition Clarity

Do not forget to recommend:
- what happens when the product becomes launched
- which page remains the review/work surface after launch
- how to avoid making launch feel “done” when some work still remains

### 4. Founder-Friendly Framing

Keep guidance practical:
- tell the user the next correct step
- explain why it matters
- avoid vague “keep iterating” advice

## Output Style

When using this skill for user-facing guidance:
- separate **critical before launch** from **can finish after launch**
- explain *why* each item matters in one sentence
- keep the language calm and non-alarmist
- make the next action obvious

## Preferred Priority Structure

### Critical Before Launch
Items that must be fixed before release.

### Important, Not Blocking
Items that improve quality but should not delay launch.

### After Launch
Items that remain visible and should continue to be completed.

## Common Launch-Readiness Heuristics

Flag these as high-risk when advising users:
- declaring launch ready while blockers still exist
- hiding unfinished non-critical items after launch
- treating checklist completion as the same thing as launch readiness
- mixing readiness advice with store compliance or analytics setup

## How Tiramisup Should Use This Skill

If Tiramisup is generating guidance for users, prefer outputs like:
- launch readiness summary
- blocker-first checklist review
- “what still remains after launch” card
- stage transition explanation

Do not present this as an absolute go/no-go judgment. Present it as practical launch readiness guidance.
